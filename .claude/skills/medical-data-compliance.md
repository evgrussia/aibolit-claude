# Skill: Medical Data Compliance

> Регуляторное соответствие: 152-ФЗ, ФЗ-323, медицинская тайна, информированное согласие

## Назначение

Обеспечение соответствия Aibolit AI российскому законодательству о персональных данных (152-ФЗ), медицинской деятельности (ФЗ-323) и смежным нормативным актам.

  Статья 11: Биометрические ПД
  Статья 13: Особенности обработки ПД в ИС

    - Явное согласие субъекта на обработку специальных категорий
    - Уведомление Роскомнадзора
    - Назначение DPO (ответственного за обработку ПД)
    - Локализация данных на территории РФ

ФЗ-323 "Об основах охраны здоровья граждан":
  Статья 13: Врачебная тайна
  Статья 20: Информированное добровольное согласие
  Статья 22: Информация о состоянии здоровья
  Требования:
    - Соблюдение врачебной тайны
    - Получение ИДС перед медицинским вмешательством
    - Право пациента на информацию о здоровье

Приказ Минздрава 834н:
  - Порядок ведения медицинской документации
  - Сроки хранения: 25 лет (медкарта), 5 лет (анализы)

ГОСТ Р 57580.1-2017:
  - Безопасность финансовых операций (для платежей за подписку)

Приказ ФСТЭК №21:
  - Требования к защите ПД в информационных системах
  - Уровни защищённости (УЗ-1, УЗ-2, УЗ-3, УЗ-4)
  - Для медицинских данных: минимум УЗ-2

## Категории данных

### Классификация

```yaml
Обычные ПД:
  примеры: [ФИО, email, телефон, дата рождения, адрес]
  основание: Согласие субъекта (ст.6 152-ФЗ)
  защита: Стандартная (УЗ-3)

Специальные категории ПД (медицинские):
  примеры:
    - Диагнозы и коды МКБ-10
    - Результаты анализов и обследований
    - История болезни
    - Медицинские изображения
    - Генетические данные
    - Информация о репродуктивном здоровье
    - Психиатрический анамнез
    - ВИЧ-статус
    - Информация о зависимостях
  основание: Явное согласие (ст.10 152-ФЗ) + ИДС (ст.20 ФЗ-323)
  защита: Повышенная (УЗ-2)

Врачебная тайна (ст.13 ФЗ-323):
  включает:
    - Факт обращения за медицинской помощью
    - Состояние здоровья
    - Диагноз
    - Сведения, полученные при обследовании и лечении
  раскрытие_без_согласия:
    - Угроза жизни пациента (экстренные состояния)
    - Расследование несчастного случая
    - Запрос суда/следствия
    - Обязательное информирование (инфекции)
```

## Информированное согласие

### Формы согласия

```yaml
1. Согласие на обработку ПД (152-ФЗ):
   Обязательные поля:
     - ФИО субъекта
     - Цель обработки
     - Перечень обрабатываемых данных
     - Перечень действий с данными
     - Срок действия согласия
     - Порядок отзыва
   Формат: Электронное (с подтверждением в Telegram)
   Хранение: 75 лет (весь срок обработки + 5 лет)

2. Согласие на AI-консультацию:
   Содержание:
     - AI-система НЕ заменяет врача
     - Результаты носят информационный характер
     - Пациент осведомлён об ограничениях AI
     - Согласие на обработку медицинских данных AI-системой
   Формат: Электронное (чекбокс при первом использовании)
   Обновление: При существенных изменениях системы

3. Согласие на обработку медицинских изображений:
   Содержание:
     - Изображения обрабатываются AI-моделью
     - Результат — предварительный скрининг, не диагноз
     - Данные хранятся в зашифрованном виде
     - Срок хранения и порядок удаления
   Формат: Электронное (перед первой загрузкой изображения)
```

### Отзыв согласия

```yaml
Процесс:
  1. Пациент запрашивает отзыв (через бота/личный кабинет)
  2. Система подтверждает получение запроса
  3. В течение 30 дней:
     - Прекращение обработки данных
     - Деперсонализация или удаление данных
     - Сохранение аудит-логов (обезличенных)
  4. Уведомление пациента о выполнении
  5. Логирование: consent_revoked

Исключения (данные НЕ удаляются):
  - Обезличенные статистические данные
  - Данные, необходимые по закону (минимальный retention)
  - Аудит-логи (обезличенные)
```

## Технические требования защиты

### Шифрование

```yaml
At Rest:
  algorithm: AES-256-GCM
  key_management: Django Fernet (development) → HashiCorp Vault (production)
  scope:
    - Все поля медицинских данных в БД
    - Медицинские изображения в файловом хранилище
    - Backup'ы базы данных

In Transit:
  protocol: TLS 1.3 (минимум TLS 1.2)
  certificates: Let's Encrypt (auto-renewal)
  HSTS: Enabled (max-age: 31536000)

Application Level:
  - Медицинские поля модели: EncryptedTextField / EncryptedCharField
  - Маскирование в логах: автоматическое (settings.AUDIT_MASKED_FIELDS)
  - API responses: медицинские данные только авторизованному пациенту
```

### Деперсонализация

```yaml
Методы:
  k-anonymity:
    description: "Каждая запись неотличима от k-1 других"
    application: "Аналитика использования AI-агентов"
    minimum_k: 5

  pseudonymization:
    description: "Замена идентификаторов псевдонимами"
    application: "Внутренняя аналитика, training data"
    reversible: true (ключ хранится отдельно)

  anonymization:
    description: "Необратимое удаление идентифицирующих данных"
    application: "Публичная статистика, исследования"
    reversible: false

Когда применять:
  - Аналитика: pseudonymization минимум
  - ML training: anonymization
  - Исследования: anonymization
  - Backup retention > 5 лет: anonymization
```

## Сроки хранения данных

```yaml
Категория → Срок:
  Медицинские карты: 25 лет (Приказ 834н)
  AI-консультации: 5 лет
  Результаты анализов: 5 лет
  Медицинские изображения: 5 лет
  Согласия пациентов: 75 лет (весь срок + 5 лет)
  Аудит-логи (медицинские): 5 лет
  Аудит-логи (общие): 90 дней
  Деперсонализированная аналитика: Бессрочно
  Платёжные данные: 5 лет (ФЗ о бухучёте)

Автоматическая очистка:
  schedule: "0 3 * * *" (ежедневно в 03:00)
  process:
    1. Найти записи с истёкшим retention
    2. Деперсонализировать (если нужна статистика)
    3. Удалить оригинальные данные
    4. Логировать: data_retention_cleanup
```

## Аудит и контроль доступа

### Доступ к медицинским данным

```yaml
Принцип: Need-to-know + Least privilege

Роли:
  patient:
    - Свои данные: полный доступ (чтение)
    - Чужие данные: запрещено
    - Удаление: по запросу (через отзыв согласия)

  ai_agent:
    - Данные текущей консультации: чтение
    - Исторические данные: чтение (с согласия)
    - Запись: результаты анализа, рекомендации
    - Без доступа к данным других пациентов

  doctor (при эскалации):
    - Данные переданного кейса: полный доступ
    - Запись: врачебные рекомендации
    - Без доступа к не-назначенным кейсам

  admin:
    - Обезличенная аналитика: доступ
    - Персональные данные: ЗАПРЕЩЕНО (кроме DPO)
    - Аудит-логи: чтение

  dpo (Data Protection Officer):
    - Аудит обработки ПД
    - Обработка запросов субъектов
    - Контроль соответствия
```

### Аудит-лог

```yaml
Что логировать:
  - Каждый доступ к медицинским данным (кто, когда, к чему)
  - Изменения медицинских данных
  - Согласия и их отзывы
  - Экспорт/скачивание данных
  - Попытки несанкционированного доступа
  - Деперсонализация и удаление данных

Формат: AuditLogService.log_medical() (см. rules/04-logging.md)
Retention: 5 лет
Неизменяемость: Append-only (логи нельзя редактировать/удалять)
```

## Чеклист для Compliance Agent

```yaml
При аудите проверять:
  Данные:
    - [ ] Все медицинские данные зашифрованы (at rest + in transit)
    - [ ] Маскирование в логах работает
    - [ ] Деперсонализация при аналитике
    - [ ] Retention policies настроены и работают
    - [ ] Backup'ы зашифрованы

  Согласия:
    - [ ] ИДС получается перед AI-консультацией
    - [ ] Согласие на обработку ПД получено
    - [ ] Механизм отзыва согласия работает
    - [ ] Согласия хранятся с правильным retention

  Доступ:
    - [ ] RBAC настроен корректно
    - [ ] Пациент не видит чужие данные
    - [ ] AI-агент не имеет доступа к данным других пациентов
    - [ ] Аудит-лог фиксирует все обращения

  Процедуры:
    - [ ] DPO назначен
    - [ ] Уведомление Роскомнадзора отправлено
    - [ ] Политика конфиденциальности опубликована
    - [ ] Процедура реагирования на утечку определена
    - [ ] Регулярный аудит безопасности запланирован
```
